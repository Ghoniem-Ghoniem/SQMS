Readability

1         

Write comments on top of all methods to describe their usage and expected input types and return type information.

 

 

2         

Avoid using short variable names

Cn ->Connection

Str->SQLQueryString

 

 

3         

Avoid using non-meaning variable names

TextBox1 ->txtShortName

 

 

4         

Define the variable before you use it directly to have concrete blocks of code.

 

 

5         

Static variables should describe why they are declared static.

 

 

6         

Complex algorithms should be thoroughly commented.

 

 

7         

Code that has been optimized or modified to "work around" an issue should be thoroughly commented, so as to avoid confusion and re-introduction of bugs.

 

 

8         

Code that has been "commented out" should be explained or removed.

 

 

9         

Code that needs to be reworked should have a TODO comment and a clear explanation of what needs to be done.

 

 

10     

Don't pass any boolean flag to a function in order to direct the business (SRP). Instead use two different functions for the negative and positive states

 

 

11     

In order to increase usability, abstract any function that may be called from multiple classes

 

 

12     

Maximum number of arguments to a function should be less than or equal 4

 

 

13     

Functions should start with a verb.

 

 

14     

Dont import anything thats not needed

 

 

15     

Don't comment code just remove it. You can go back by checking out a previous version from SVN

 

 

16     

Use descriptive names for functions and variables

 

 

17     

Be sure that you don't make any spelling mistake (to be able to search easily)

 

 

18     

Naming for the variable, member function and class should represent the business need

 

 

19     

All codes should be separated in suitable functions, not directly under the controller [Event handler].

 

 

20     

Use enumerations for groups of related constants that have integer values.

 

 

Usability

1         

Extract a method if the same piece of code is being used more than once or you expect it to be used in future.

 

 

2         

Make some generic methods for repetitive task and put them in a related class so that other developers start using them once you intimate them.

 

 

3         

Use interfaces wherever needed to maintain decoupling.

 

 

4         

For strings and localization you should put all strings in application resources file.

 

 

5         

Use access specifiers (private, public, protected, internal, protected internal) as per the scope need of a method, a class, or a variable.

 

 

6         

Interfaces should be as small as possible.

 

 

Code and IDE performance

1         

Make sure that there shouldn't be any project warnings.

 

 

2         

All unused usings need to be removed. Code cleanup for unnecessary code is always a good practice.

 

 

3         

Disposing of Unmanaged Resources like File I/O, Network resources, etc. They have to be disposed of once their usage is completed. Use usings block for unmanaged code, if you want to automatically handle the disposing of objects once they are out of scope.

 

 

4         

Avoid nested for/foreach loops and nested if conditions as much as possible.

 

 

5         

Use a Stringbuilder instead of string if multiple concatenations are required, to save heap memory.

 

 

6         

Avoid type casting and type conversions as much as possible; because it is a performance penalty.

 

 

7         

If you have many items in branching code you should definitely use a switch, not if else.

 

 

8         

When you use (If/Else) try to put the most likely true conditions at the first.

 

 

9         

If you have complex condition in branching you should use (IF/Else).

 

 

10     

Arrange your if conditions with setting the most probability at the first

 

 

11     

Never use .ToString with string objects

 

 

12     

Never use redundant qualifier

This.MyText;

 

 

13     

Never initialize the variable by its default value

 

 

14     

Never run or build your application in debug mode, just use debug mode when debugging.

 

 

15     

 

 

 

16     

For parameters casting used through methods, it should be done only once at the first of method.

 

 

Responsibility

 

 

 

 

1         

Every class, operation should have only one responsibility.(SRP)

 

 

2         

No one can access the variables except throw access modifiers or functions in order to control the call stack

 

 

 

Maintainability

 

1         

All resources, images, icons strings, localization info should be separated in different and unified resource file.

 

 

 

2         

Do not drill down in a reference many times, get the value and leave the reference.

 

 

 

3         

If you have a sequence of calling for a reference, and this operation is repeated over the code, move this operation to a standalone method.

Ex: Formatting the resources from resourcelib.

 

 

 

4         

Avoid bloated code like:

Something something = new Something();

something.setField1(value1);

something.setField2(value2);

something.setField3(value3);

 

it should be :

Something something = new Something(value1, value2, value3);
something.doSomething();

 

 

 

5         

Function should not exceed 30 lines of codes.

 

 

 

6         

Avoid copy and paste for code bulks to finalize your jobs.

 

 

 

7         

Use panels and groups in order to enable and disable group of related attributes

 

 

 

8         

Never duplicate the same bulk of code, use separate function to perform the related operation.

 

 

 

9         

Never duplicate the behavior according to the language.

 

 

 

10     

Class diagram should represent the business for variables and operations.

 

 

 

11     

Avoid uncalled members and classes.

 

 

 

12     

Do not use qualifier while no need for it:

oldFont = new System.Drawing.Font(Font.FontFamily, Font.Size, Font.Style, Font.Unit);

 

 

 

13     

Method parameters should be primitive as much as possible in order to facilitate migration to component based architecture simply

 

 

 

14     

Delete unreached codes

 

 

 

15     

Delete unused namespaces, classes, variables

 

 

 

16     

Do not initialize variable without need for validation on its value before generating the real object.

 

 

 

17     

Function parameters should be limited to what will be used in the function scope

protected void SetViewMode(bool bCancelEditing, int nNewID)

{

IsUpDownPressed = 0;

CancelEditNode = bCancelEditing;

}

 

 

 

 

Defensive programming

 

1         

Check for all values that should not be empty

 

 

 

2         

Check for negative and positive values according to your needs

 

 

 

3         

Check for null objects before usage

 

 

 

4         

Check for connection status, before using it

 

 

 

5         

Check for object validity for usage.

 

 

 

6         

Write logs for all exceptions or main functionality success events with all its parameters:

-          Date time

-          Inputs

-          Outputs

 

 

 

7         

Check for function inputs validity before execution.

 

 

 

8         

'Null' check needs to be performed wherever applicable to avoid the Null Reference Exception at runtime.

 

 

 

9         

Check for DB connection status before usage

 

 

 

10     

Check for web Service connection status before usage

 

 

 

 

Tractability

1         

Log for the execution as follow:

Module, class, operation, start parameters, execution steps, success flag, exception description.

 

 

2         

Logging should have suitable retention period in order to prevent function tractability or increase of logging media

 

 

3         

Logging should be in separate module in order to be used from all components

 

 

4         

Logging should be found in all building blocks found in the deployment model.
